# Coach Quick Registration Page
After user has provided valid PIN code or password in Coach Login Dialog, the Coach Quick Registration Page is shown. This page shows training sessions of two weeks prior and two weeks forward from current day.

## Altered Sheet Schemas

### Sessions sheet
- `sessions` sheet is for storing the time span training sessions are available

| Column | Name               | Type     |
|--------|--------------------|----------|
| A      | id                 | string   |
| B      | session_type       | string   |
| C      | session_type_alias | string   |
| D      | start_date         | ISO-8601 |
| E      | end_date           | ISO-8601 |
| F      | created_at         | ISO-8601 |
| G      | updated_at         | ISO-8601 |

#### Column explanations
- A: unique id (UUID) generated by backend when appending row into the sheet
- B: session type is used for storing English version of the session name
- C: session type alias is used for storing the session name in some other language (in this case Finnish)
- D: start date of the session in format 'YYYY-MM-DD' (for example: '2026-01-03')
- E: end date of the session in format 'YYYY-MM-DD' (for example: '2026-12-31')
- F: created timestamp when the row was appended (backend should set this)
- G: updated timestamp when the row was updated last time (backend should set this when row is updated)

## New Sheet Schemas

### Weekly Schedule sheet
- `weekly_schedule` sheet is used for storing sessions weekly schedule that are available for registration

| Column | Name               | Type     |
|--------|--------------------|----------|
| A      | id                 | string   |
| B      | session_type	      | string   |
| C      | weekdays_available | number   |
| D      | start_time	        | time     |
| E      | end_time           | time     |	
| F      | location           | string   |	
| G      | active             | boolean  |
| H      | created_at         | ISO-8601 |
| I      | updated_at         | ISO-8601 |

#### Column explanations
- A: unique id (UUID) generated by backend when appending row into the sheet
- B: session type is used for storing English version of the session name (matches the value of session_type in `sheets` sheet)
- C: weekday when the training session is available in number format
    * 0 = Monday, 1 = Tuesday, 2 = Wednesday, 3 = Thurday, 4 = Friday, 5 = Saturday, 6 = Sunday
    * May contain combination of days (numbers) for example: 1,3 (meaning this type of session is held Tuesdays and Thursdays)
- D: start time of the session indicating at what time the session starts on designated day(s) in format 'HH:MM' (for example: '18:30')
- E: end time of the session indicating at what time the session end on designated day(s) in format 'HH:MM' (for example: '18:30')
- G: is session or sessions active or not at the moment
- H: created timestamp when the row was appended (backend should set this)
- I: updated timestamp when the row was updated last time (backend should set this when row is updated)

#### Column explanations

### Camp Schedules sheet
- `camp_schedules` sheet is used to store camp sessions data

| Column | Name                  | Type     |
|--------|-----------------------|----------|
| A      | id                    | string   |
| B      | camp_id               | string   |
| C      | session_name          | string   |
| D      | date                  | ISO-8601 |
| E      | start_time	           | time     |
| F      | end_time 	           | time     |
| G      | created_at            | ISO-8601 |
| H      | updated_at            | ISO-8601 |

#### Column Explanations
- A: unique id (UUID) generated by backend when appending row into the sheet
- B: id of the camp (from `camps` sheet's id field) connects a session to a certain camp e.g. foreign key
- C: name of the session
- D: date when the session is held in format 'YYYY-MM-DD' (for example: '2026-01-03')
- E: start time of the session indicating at what time the session starts on designated day in format 'HH:MM' (for example: '18:30')
- F: end time of the session indicating at what time the session end on designated day in format 'HH:MM' (for example: '18:30')
- G: created timestamp when the row was appended (backend should set this)
- H: updated timestamp when the row was updated last time (backend should set this when row is updated)

### Coach Registrations sheet
- `coach_registrations` sheet is used for storing coach registrations for sessions.

| Column | Name                  | Type     |
|--------|-----------------------|----------|
| A      | id                    | string   |	
| B      | first_name            | string   |	
| C      | last_name	           | string   |	
| D      | session_type	         | string   |
| E      | date	                 | ISO-8601 |
| F      | realized	             | boolean  |
| G      | start_time            | time     |
| H      | end_time              | time     |
| I      | created_at            | ISO-8601 |
| J      | updated_at            | ISO-8601 |

#### Column Explanations
- A: unique id (UUID) generated by backend when appending row into the sheet
- B: firstname of the coach
- C: lastname of the coach
- D: session type (much match one of the sessions' types in `sessions` sheet)
- E: the date when the session is being held or was held
- F: was the session actually held or not
- G: start time of the session in format 'HH:MM' (for example: '18:30'). This is must only for free/sparring sessions
- H: end time of the session in format 'HH:MM' (for example: '18:30'). This is must only for free/sparring sessions
- I: created timestamp when the row was appended (backend should set this)
- J: updated timestamp when the row was updated last time (backend should set this when row is updated)

## Operation flow

### Preconditions
User must have provided a valid PIN code or password in Coach Login Dialog

### Steps
1. Setting value to 'Logged in as: ' field (use translation key)
    - If the user used PIN code to identify him/herself the name of the coach is set into 'Logged in as: ' field
    - If user logged in using the password 'Logged in as: ' field shows text 'You are not registered' (use translation key)
2. Application makes a request to backend via GAS API to fetch training sessions info.
    - Application shows loading spinner and blocks user interaction with UI until response is received from the backend
    - GAS read all data from `sessions`, `weekly_schedule` and `coach_registrations` sheets
    - Filters `coach_registrations` data to only include those with `realized` column status not FALSE (handles boolean and string values robustly).
    - Builds a map of session types to their active periods (start/end dates) from the sessions sheet.
    - Calculates a 21-day window starting from seven days prior the current week’s Monday.
        - For each date in the window:
            - Iterates through `weekly_schedule` rows, checking if the session is active and matches the weekday.
            - Checks if the session type is active for the date.
            - Formats start/end times.
            - Finds registered coaches for the session and date.
            - Adds session objects to the result array.
        - Additionally, for each date, finds `free/sparring` session types from the `coach_registrations` sheet with a designated coach and realized status not FALSE and add into result array.
            - Adds these as separate session objects.
    - Reads data from `camps` sheet 
        - Replaces overlapping sessios with sessions from `camp` and `camp_schedules` sessions
    - Sorts the sessions by date and start time.
3. Cards are built from received session objects.
    - shows the date and time (start time and end time from `weekly_schedule` or `camp_schedule`), session type (when English localization) or session type alias (when other localization is used), location, and assigned coach (if any)
    - cards are grouped by date
    - if session do not have designated coach 
        - the card background color is set to theme's accent secondary 
        - a card action's button with text 'Register' is shown (use translation key)
    - if session have designated coach 
        - the card background color is set to light green
        - a card action's button with text 'Remove' is shown (use translation key)

### Buttons

### Page Buttons 
- Clicking 'Refresh data' (use translation key) button refreshes the data shown on page by requesting fresh data from backend
- Clicking 'Back to main' (use translation key) button clears logged in coach's data state (logs out) and navigates the user to Main Page

#### Card Action Buttons
- When 'Register' button clicked then show dummy Confirm Coach Registration dialog
- When 'Remove' button is clicked then show dummy Confirm Remove Coach From Session dialog

## Error Handling
- If backend request fails, show error message (use translation key).
- If no sessions are available, show 'No sessions found' (use translation key).

## Acceptance Criteria
- Sessions are shown for 2 weeks prior and 2 weeks forward from today.
- MUI components are used
- the colors are used in components where applicable
- Cards are grouped by date.
- Register/Remove buttons trigger confirmation dialogs.
- All labels and buttons use correct translation keys.
- UI handles empty data and backend errors gracefully.
